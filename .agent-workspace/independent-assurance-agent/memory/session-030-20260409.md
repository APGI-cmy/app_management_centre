## Learning Notes

- CORR-001/002 (git ls-tree and SHA placeholders) are a recurring pattern in §4.3c sequences where the producing agent commits the PREHANDOVER proof before the commit SHA is known, then must correct. Consider whether pre-brief should explicitly state that SHA fields are MANDATORY to populate in a follow-up commit before IAA invocation, and that IAA will verify these are populated.
- CORR-003 (canon entry count discrepancy) arose because multiple parallel waves merged between the layer-down branch creation and the ECAP-001 merge. The CANON_INVENTORY count in PREHANDOVER proofs should reference both branch-state count (at wave execution) AND merged-state count (at PR merge) to avoid post-merge confusion.
- CORR-005 pattern (token file missing at time of merge) is preventable: the pre-brief should state that the IAA token file creation is a hard prerequisite before PR opens, not after IAA verbal approval.

---

## Suggestions for Improvement

1. **Pre-brief enhancement**: Add explicit requirement that §4.3c SHA fields (`git ls-tree HEAD`, `Commit SHA`) must be populated via a follow-up commit before IAA is invoked. Zero placeholders should be enforceable at pre-IAA invocation gate, not discovered post-merge.

2. **CANON_INVENTORY count reporting**: Producing agents should record both the branch-state canon count AND the expected merged-state count in PREHANDOVER proofs for layer-down waves, to prevent the CORR-003 class of discrepancy.

3. **Token file creation gate**: The pre-brief should explicitly state that the dedicated IAA token file (`.agent-admin/assurance/iaa-token-session-NNN-waveY-YYYYMMDD.md`) must be written before the PR is opened. Current flow creates the token during IAA review, which means the PR branch is missing the token at the time it is opened. The First Invocation Exception handles this architecturally but the ceremony is cleaner if the token exists at PR open time.
