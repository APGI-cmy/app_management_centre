## Learning Notes

**LEARN-021-001**: Pre-PR invocation waiver mechanism gap. The IAA_ZERO_SEVERITY_TOLERANCE.md §Exception Procedure requires CS2 to post a comment on the specific PR, but pre-PR invocations make this impossible (PR not open). Practical resolution: CS2-directed issue authority accepted as equivalent. This gap should be addressed in a governance update to IAA_ZERO_SEVERITY_TOLERANCE.md — add a pre-PR invocation exception clause allowing the CS2-authorized issue to serve as the waiver when the PR has not yet been opened.

**LEARN-021-002**: A-036 design confirmation. The invocation-discipline repeat check (A-036) is NOT APPLICABLE when the current invocation is clean — it only triggers when the current invocation has an ENVIRONMENT_BOOTSTRAP failure that matches a prior session. This is correctly scoped: A-036 is a "repeat check" (catches patterns across sessions), not a retroactive punishment for prior sessions that have been resolved.

**LEARN-021-003**: FAIL-ONLY-ONCE.md version header discipline. When adding new A-rules to FAIL-ONLY-ONCE.md, the version header should be bumped AND a new version history table entry added. This was not done for A-036. Future invoking agents should be reminded of this hygiene requirement when adding to FAIL-ONLY-ONCE.md. Could be enforced via a new CORE check or OVL-KG check if this pattern recurs.

**LEARN-021-004**: CodexAdvisor session numbering vs. IAA session numbering creates potential confusion in token file naming. In this session, CodexAdvisor used session-015 as the token reference (IAA-session-015-*) while IAA's own sequential numbering would give session-021. The A-029 architecture requires the PREHANDOVER's iaa_audit_token to match the actual token file. Future: CodexAdvisor should pre-populate iaa_audit_token with IAA's next session number, not its own. Flagged as follow-up improvement.
