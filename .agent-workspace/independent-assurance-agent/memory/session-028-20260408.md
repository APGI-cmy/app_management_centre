The PREHANDOVER proof's `iaa_audit_token` field is frozen at commit time to an expected session number, but if that session issues a REJECTION-PACKAGE, the next invocation uses a different session number. This is not currently documented as an expected/handled scenario. Recommend adding to FAIL-ONLY-ONCE.md or AGENT_HANDOVER_AUTOMATION.md: "When a REJECTION-PACKAGE is issued, the re-invocation will use a new session number. The PREHANDOVER proof iaa_audit_token field remains as the original expected reference (read-only). The new token file takes precedence. This is not a ceremony failure." This would prevent future auditors from flagging the session-number mismatch as a CORE-019 issue.
