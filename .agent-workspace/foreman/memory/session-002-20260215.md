### What Future Sessions Should Know
- **Governance ripple has TWO workflows**: 
  - `governance-ripple-sync.yml` (event-driven, primary) 
  - `governance-alignment.yml` (scheduled, fallback)
  - Both must have feature parity (especially auto-merge)

- **Dispatch success != PR creation success**: 
  - Canonical dispatch logs show "✅ Dispatched successfully" 
  - Does NOT mean receiver created PR
  - Requires checking receiver workflow runs

- **Consumer registry is authoritative**:
  - Located in canonical repo: `governance/CONSUMER_REPO_REGISTRY.json`
  - Lists all consumer repos and enabled status
  - Must be checked when investigating "why didn't repo X get ripple?"

- **Auto-merge requires specific pattern**:
  - Use `gh pr merge --auto --squash --delete-branch`
  - Requires PR number from `gh pr list`
  - Must wait briefly (sleep 2) after PR creation for API consistency

### Governance Insights
- **Insight 1**: Governance ripple is a complex distributed system with multiple failure points
  - Canonical dispatch (can fail at source)
  - Network delivery (GitHub repository_dispatch API)
  - Receiver workflow (must exist, be enabled, and be configured correctly)
  - Alignment script (must detect drift and stage changes)
  - PR creation (must succeed and enable auto-merge)
  - Merge gates (must all pass for auto-merge to proceed)

- **Insight 2**: Scheduled fallback (governance-alignment.yml) is critical safety net
  - Runs hourly to catch missed ripples
  - Should have same capabilities as event-driven workflow
  - Current issue: Only fallback has auto-merge (now fixed in ripple-sync too)

- **Insight 3**: Investigation issue templates are powerful delegation mechanism
  - Provides structured checklist for local agents
  - Respects authority boundaries (doesn't require cross-repo push access)
  - Enables parallel investigation without Foreman bottleneck
