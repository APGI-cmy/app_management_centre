### What Future Sessions Should Know

1. **Always check Actions logs**: Repository might be receiving events even if not visible
2. **Version-only drift detection is insufficient**: File-level SHA256 verification required for accuracy
3. **ISMS is reference implementation**: When in doubt, compare with ISMS for governance patterns
4. **Workflow naming matters**: `governance-ripple-sync.yml` vs `governance-alignment.yml` signals different purposes

### Governance Insights

1. **Drift detection strategies**: Version comparison (fast) vs SHA256 verification (accurate) serve different needs
2. **Event-driven + scheduled**: Complementary strategies provide both immediate response and safety net
3. **Workflow harmonization**: Multiple consumer repos should use consistent patterns for canonical governance alignment
4. **Evidence is key**: Actions logs proved ripple was working, narrowing investigation to processing logic

### Technical Patterns Learned

1. **Repository dispatch debugging**: Check Actions history with `--event repository_dispatch` flag
2. **Workflow comparison**: Side-by-side YAML comparison reveals subtle but critical differences
3. **Script reuse**: Both repos have `align-governance.sh` but only ISMS workflow calls it
4. **PR automation**: `peter-evans/create-pull-request@v6` action provides robust PR creation
